Step 1.2 — Chunk text with overlap
Roadmap's Month 2 phase already covers "chunk size and overlap" — reuse that exact approach here. Chunks that are too small lose context, too large waste tokens and hurt retrieval precision.

Step 2.1 — Try pdfplumber table extraction first
Roadmap names both pdfplumber AND Camelot for tables — they're complementary, not interchangeable. pdfplumber is faster and works for simple grid tables; Camelot handles complex/borderless tables pdfplumber misses.
pdfplumber.extract_tables()
Gemini prompt — paste exactly
In src/ingestion/table_extractor.py, create extract_tables_pdfplumber(pdf_path: str) -> list[dict]. For each page, call page.extract_tables(). For each table found, return {"page_number": int, "rows": list[list[str]], "method": "pdfplumber"}. Filter out tables with fewer than 2 rows or fewer than 2 columns — those are likely false positives, not real tables. Log how many tables were found per page.
Step 2.2 — Camelot fallback for pages with no pdfplumber tables
Only runs on pages where Step 2.1 found nothing — Camelot is slower so we don't waste time running it on every page.
camelot-pyflavor="lattice" and "stream"
Gemini prompt — paste exactly
In src/ingestion/table_extractor.py, create extract_tables_camelot(pdf_path: str, page_numbers: list[int]) -> list[dict]. For each page number given, try camelot.read_pdf(pdf_path, pages=str(page_number), flavor="lattice") first. If that returns zero tables, retry with flavor="stream". For each table found, convert table.df to a list of lists and return {"page_number": int, "rows": list[list[str]], "method": "camelot"}. Runs extract_tables_pdfplumber on all pages 2. Finds which pages had zero results 3. Runs extract_tables_camelot only on those pages 4. Returns combined results from both methods

Phase 4 — Hybrid search
Vector similarity search · BM25 keyword search · combined ranking
~2 days ▾
Step 4.1 — Vector similarity search function
Pure semantic search — finds chunks that mean something similar to the question, even if exact words don't match.
Gemini prompt — paste exactly
In src/retrieval/vector_search.py, create vector_search(query: str, document_id: str, top_k: int = 10) -> list[dict]. 1. Call embed_text(query) to get the query embedding 2. Run a SQL query using pgvector's cosine distance operator (<=>) against DocumentChunk where document_id matches, ordered by distance ascending, limit top_k 3. Return list of {"chunk_id": str, "content": str, "page_number": int, "chunk_type": str, "score": float} where score = 1 - distance Use raw SQL via SQLAlchemy text() for the vector query since pgvector's operator needs special handling.
Step 4.2 — BM25 keyword search function
Catches exact terms vector search can miss — specific numbers, names, codes. Critical for financial reports where "$4.2M" needs to match exactly.
rank-bm25
Gemini prompt — paste exactly
In src/retrieval/bm25_search.py, create a class BM25Index with: 1. __init__(self, document_id: str): loads ALL DocumentChunk rows for that document_id from the database, tokenizes each chunk's content by lowercasing and splitting on whitespace, builds a BM25Okapi index from rank_bm25 library, stores chunk metadata in self.chunk_map keyed by index position 2. search(self, query: str, top_k: int = 10) -> list[dict]: tokenizes the query the same way, calls self.bm25.get_scores(tokenized_query), returns top_k results sorted by score descending as {"chunk_id": str, "content": str, "page_number": int, "chunk_type": str, "score": float} Note: this index is rebuilt on each call for now — caching/persistence comes later if needed. Keep it simple and correct first.
Step 4.3 — Combine both into hybrid search
Roadmap explicitly says "hybrid search combining vector similarity and BM25" — this is the exact combination step, using a standard rank-fusion approach so neither method dominates unfairly.
Gemini prompt — paste exactly
In src/retrieval/hybrid_search.py, create hybrid_search(query: str, document_id: str, top_k: int = 10) -> list[dict]. 1. Call vector_search(query, document_id, top_k=20) and bm25_search via BM25Index(document_id).search(query, top_k=20) 2. Combine using Reciprocal Rank Fusion: for each chunk_id, score = 1/(rank_in_vector_results + 60) + 1/(rank_in_bm25_results + 60) — if a chunk only appears in one list, only that term counts 3. Sort all unique chunks by combined score descending 4. Return top_k results with their original content/page_number/chunk_type plus the fused score Add a parameter retrieval_method: str = "hybrid" that can also accept "vector" or "bm25" to run just one method alone — useful for testing/comparison later.
Deliverable check
✓ Searching an exact phrase from the PDF returns that chunk even via BM25 alone
✓ Searching a paraphrased question returns relevant chunks via vector search alone
✓ hybrid_search returns better-ranked results than either method alone for at least 3 test questions
Phase 5 — Self-querying retrieval
Gemini rewrites the question · extracts filters · then searches
~1 day ▾
Step 5.1 — Gemini rewrites the user's question
Roadmap says "self-querying retrieval where LLM rewrites the query before searching." Users ask vague things like "how did we do last year" — Gemini turns that into a search-optimized version plus structured filters (like a page range or chunk type hint) before hybrid_search runs.
Gemini prompt — paste exactly
In src/retrieval/self_query.py, create rewrite_query(user_question: str) -> dict. Send this EXACT prompt to Gemini (gemini-1.5-flash): --- SYSTEM --- You rewrite user questions into better search queries for a document retrieval system over an annual report. Respond ONLY with valid JSON, no markdown, no explanation. Return exactly: { "search_query": string (a clear, specific rewrite of the question, expanding vague terms — e.g. "last year" should be left as-is since you don't know the actual year, but vague pronouns should be clarified), "preferred_chunk_type": "text" | "table" | "image_caption" | "any" (guess which type of content likely contains the answer — numbers/financials suggest "table", trends/visuals suggest "image_caption", general questions suggest "any") } --- USER --- Question: {user_question} --- Parse and return the JSON. On parse failure, return {"search_query": user_question, "preferred_chunk_type": "any"}.
Step 5.2 — Wire self-query into hybrid search
Connects the rewrite step to retrieval — if Gemini guessed a specific chunk_type, results are boosted toward that type rather than strictly filtered, so we don't accidentally exclude the real answer.
Gemini prompt — paste exactly
In src/retrieval/hybrid_search.py, add search_with_self_query(user_question: str, document_id: str, top_k: int = 10) -> list[dict]: 1. Call rewrite_query(user_question) to get {search_query, preferred_chunk_type} 2. Call hybrid_search(search_query, document_id, top_k=20) 3. If preferred_chunk_type != "any": boost scores of chunks matching that type by multiplying their score by 1.3 (do not filter out other types completely — just re-rank) 4. Re-sort by adjusted score, return top_k This is the main function the API endpoint will call — not hybrid_search directly.
Deliverable check
✓ Asking "what was the revenue trend" returns table/image chunks ranked higher than plain text chunks
✓ Asking a general question still returns reasonable results (preferred_chunk_type doesn't break things)
